# 将想法通过头脑风暴转化为设计

## 概述

通过自然的协作对话，帮助将想法转化为完全形成的设计和规范。

首先了解当前的项目背景，然后一次问一个问题来完善想法。一旦你理解了你要构建的内容，分小节（200-300字）展示设计，并在每一节后检查目前看起来是否正确。

## 流程

**理解想法：**
- 首先检查当前项目状态（文件、文档、最近的提交）
- 一次问一个问题来完善想法
- 尽可能使用多项选择题，但开放式问题也可以
- 每条消息只问一个问题 - 如果一个主题需要更多探索，将其通过多个问题进行
- 专注于理解：目的、约束、成功标准

**探索方法：**
- 提出 2-3 种不同的方法及其权衡
- 以对话方式展示选项，并附上你的建议和理由
- 首先提出你推荐的选项并解释原因

**展示设计：**
- 一旦你认为你理解了你要构建的内容，就展示设计
- 将其分成 200-300 字的小节
- 在每一节后询问目前看起来是否正确
- 涵盖：架构、组件、数据流、错误处理、测试
- 如果有什么不合理的地方，随时准备回去澄清

## 设计之后

**文档：**
- 将验证后的设计写入 `docs/plans/YYYY-MM-DD-<topic>-design.md`
- 如果可用，使用 elements-of-style:writing-clearly-and-concisely 技能
- 将设计文档提交到 git

**实施（如果继续）：**
- 询问：“准备好进行实施设置了吗？”
- 使用 superpowers:using-git-worktrees 创建隔离的工作区
- 使用 superpowers:writing-plans 创建详细的实施计划

## 关键原则

- **一次一个问题** - 不要用多个问题让人不知所措
- **首选多项选择** - 在可能的情况下，比开放式问题更容易回答
- **无情地遵循 YAGNI** - 从所有设计中删除不必要的功能
- **探索替代方案** - 在确定之前总是提出 2-3 种方法
- **增量验证** - 分节展示设计，验证每一节
- **保持灵活** - 当有什么不合理时，回去澄清
